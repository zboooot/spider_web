# Spider Web 游戏 — 性能优化需求约束（给 NotebookLM）

> 本文档描述**不可妥协的游戏玩法与架构约束**。请在此基础上提出优化方案，不要建议破坏这些机制。

## 1. 游戏概述

- **平台**：浏览器 Canvas 2D，纯 JavaScript，Vite 构建，需支持 iPhone Safari
- **核心玩法**：Verlet 物理模拟的径向蜘蛛网 + 蜘蛛行走 + 投掷物粘网/挣脱/断丝 + 网完整度 HUD
- **性能症状**：iPhone 上猎物增多、挣脱频繁时 FPS 下降；FPS 面板显示 **Draw Call 不是主因**

## 2. 网状结构（不可改为简单绳/链）

蜘蛛网是 **径向 + 环向** 的网状拓扑，不是单条绳或布料：

- `segments` 条辐射线 × `depth` 圈 ≈ 粒子数 `n = segments × depth`
- 每条辐射链有 **距离约束**（相邻粒子）
- 每圈有 **环向距离约束**（跨 `segments` 偏移连接）
- 外圈闭合约束（首尾粒子）
- 部分辐射粒子被 **pin** 固定（`pinStep` 间隔）
- 约束刚度经 `tensor` 缩放；网可被 **局部破坏**（删除约束，非整网重建）

**运行时规模（典型关卡）**：
- ~660 条 `DistanceConstraint` 线段（可动态减少）
- ~700+ 总约束（含蜘蛛身体/腿约束）
- 网区为圆形，粒子径向分布，线段长度相近

## 3. 三类几何查询（当前均为暴力全量遍历）

### 3.1 粘网 `stickSystem`

**玩法**：投掷物进入网区后，沿运动路径收集「经过的网段候选点」，延迟后从历史中选一个粘住。

**必须保留的机制**：
- 基于 `prevX,prevY → pos` 的 **路径检测**（防 tunneling）
- `stickCatchRadius ≈ 18` 的捕获半径
- 径向权重 `radialRatio`（边缘 vs 中心粘附时长不同）
- 历史候选在断丝后需失效

**当前实现**：每猎物每帧对 **全部** 网约束，每约束 **9 点采样**（t=0..8/8）

### 3.2 落脚 `footSystem`

**玩法**：蜘蛛 4 条腿在网约束节点 + 线段采样点上找最佳落脚目标。

**必须保留的机制**：
- 搜索半径 `stepR=53`，理想距离 `idealDist=23`，最小距离 `minR=10`
- 腿间最小间距 `MIN_LEG_SEP=14`
- 候选包括：**存活约束上的粒子节点** + **线段上 N 等分采样点**（典型 N=3 → ~1980 采样点/帧）
- 断丝后只应使用仍连通的粒子/线段

**空间需求**：查询必须以蜘蛛躯干附近 **局部半径** 为范围，不能全局盲搜（但当前实现等价于扫全部约束+全部采样点）

### 3.3 完整度 `webIntegrity`

**玩法**：圆形网区内规则网格（`gridStep ≈ 35`），每格判断是否有网段覆盖（`coverD ≈ 22`），显示完整度百分比。

**必须保留的机制**：
- ~430 个网格格点
- 覆盖判定：格点到任意存活网段的距离 ≤ coverD
- 网被破坏后完整度应更新（可接受分帧/延迟，但玩家需感知到变化）

**当前痛点**：周期性或全量 `O(格点数 × 约束数)` ≈ 28 万次比较/次

## 4. 物理求解（Verlet / PBD）

- 每逻辑帧 `sim.frame(16)` — **16 次固定约束迭代**
- 使用 `DistanceConstraint` + `AngleConstraint`（蜘蛛关节）
- `AngleConstraint` 热路径存在 `new Vec2()` 分配
- **固定 dt** 思路（通过 `timeScale` 适配帧率，但物理迭代次数目前固定）
- 粘网/落脚/完整度查询目前在物理步 **之外** 的逻辑帧执行（这是正确的，不应 ×16）

## 5. 空间查询的硬性需求（请方案必须满足）

1. **网状线段索引**：方案必须支持 **动态增删** 线段（断丝），不是静态场景
2. **局部邻域查询**：粘网、落脚、完整度都需要「查询点附近 / 格点附近」的候选线段，**至少覆盖周围网格单元**（如 3×3 或 5×5 cell），不能漏检跨格长线段
3. **统一数据集**：三类查询针对同一 `spiderweb.constraints` 集合，应能 **共享空间索引**
4. **窄相精度**：broad phase 之后仍需：
   - 点-线段最近距离（落脚、完整度）
   - 运动路径-线段扫掠/投影（粘网 CCD）
5. **移动端单线程 JS**：主线程运行，需考虑 GC 尖刺（对象池 / 原地数学）

## 6. 不可接受的方案方向

- 把蜘蛛网改成单条 Verlet 绳或无环向连接的星形线（破坏网状玩法）
- 每帧全量 rebuild 索引（660 线段 + 频繁断丝时开销过大）
- 仅减少物理迭代次数但不优化三类查询（已知主瓶颈在查询）
- 把粘网检测放进 16× 物理子步内
- 完整度每帧全量 28 万次扫描
- 过早引入 BVH/SAP 等重型结构（除非资料证明对 ~600 均匀线段更优）

## 7. 验收目标

| 指标 | 目标 |
|------|------|
| iPhone 中档机，波5、Prey=8 | ≥ 30 fps |
| 挣脱/断丝时帧尖刺 | p95 frame time < 25ms |
| 完整度扫描 | 分帧，≤ 5000 次比较/帧 |
| 物理手感 | 16→10~12 迭代无明显劣化（若建议降低） |
| Draw Call | 非优化重点 |
